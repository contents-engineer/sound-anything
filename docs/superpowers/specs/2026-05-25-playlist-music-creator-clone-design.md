# Playlist Music Creator Clone — Design Spec

**Date:** 2026-05-25
**Status:** Approved, ready for implementation plan
**Source reference:** `https://tubemaster.app/lms/student/.../plugins/playlist-music-creator`

## 1. Goal

Build a functional clone of the TubeMaster "플레이리스트 음악 만들기" page as a standalone Next.js app. Users pick from 10 categories of musical attributes (genre, mood, vocal style, etc.); the app calls an LLM that returns:

1. A single composite "music generation prompt" (the kind a user would paste into Suno / Udio / etc.).
2. A list of 10 song concepts (each: title + 2–3 sentence concept/mood summary).

Original creative text is generated by the LLM from the user's selections — no reproduction of existing songs.

## 2. Scope

**In scope (MVP):**
- Full UI clone of all 10 option sections (functional layout, not pixel-perfect).
- Multi-select / single-select per section + free-text custom input per section.
- 1–10 minute length slider.
- Two action buttons:
  - **"단일 스타일 프롬프트 생성"** → only generates the composite music-gen prompt (no song list). Faster, cheaper. Calls `/api/generate?mode=prompt-only`.
  - **"Playlist 만들기"** → full result: prompt + 10 song concepts. Calls `/api/generate?mode=full`.
- Server-side LLM call via `/api/generate`.
- Provider-switchable LLM backend (OpenAI / Anthropic / Gemini / mock) via `AI_PROVIDER` env var.
- Result panel showing generated prompt + 10 song concepts.
- Browser-only history (last 20 generations in localStorage).

**Explicitly out of scope:**
- Image generation (album cover, source images).
- Video generation (background video).
- User authentication, points/billing UI, navigation to other pages.
- Server-side persistence / database.
- Automated test suite (manual + mock provider only for MVP).

## 3. Tech stack

- Next.js 15 (App Router) + TypeScript

## 5. Data model

```ts
// types.ts
export type Selections = {
  genre: string[]                 // multi-select
  mood: string[]                  // multi-select
  vocal: string[]                 // multi-select
  usage: string[]                 // multi-select
  instrument: string[]            // multi-select
  bpm: string | null              // single-select
  age: string | null              // single-select
  language: string | null         // single-select
  topic: string[]                 // multi-select
  lengthMin: number               // 1–10
  customInputs: Partial<Record<keyof Omit<Selections,'lengthMin'|'customInputs'>, string>>
}

export type SongConcept = {
  title: string                   // short original title (1–5 words)
  concept: string                 // 2–3 sentences describing mood, imagery, and hook idea — NOT lyric lines
}

export type GenerationResult = {
  mode: 'prompt-only' | 'full'
  prompt: string                  // single composite music-gen prompt
  songs: SongConcept[] | null     // null when mode='prompt-only', exactly 10 entries when 'full'
  generatedAt: string             // ISO timestamp
  provider: string                // "openai" | "anthropic" | "gemini" | "mock"
}
```

## 6. Data flow

1. User toggles preset buttons + types optional custom text per section → updates single React state `selections`.
2. User clicks one of the action buttons → client POSTs `{ selections, mode }` to `/api/generate` (mode = `'prompt-only'` or `'full'`).
3. API route:
   - Validates that at least one option is selected.
   - Calls `getProvider()` → reads `AI_PROVIDER` env.
   - Provider's `generate(selections, mode)` builds prompts, calls LLM, parses JSON. In `prompt-only` mode the system prompt instructs the model to return only the composite prompt with `songs: null`; in `full` mode it returns prompt plus exactly 10 song concepts.
   - Returns `GenerationResult`.
4. Client renders `<ResultPanel>` (which adapts to the `mode`) and prepends entry to localStorage history (`tubemaster-clone:history`, max 20).

## 7. Provider abstraction

```ts
// lib/ai/provider.ts
export interface AIProvider {
  name: string
  generate(opts: Selections, mode: 'prompt-only' | 'full'): Promise<GenerationResult>
}

export function getProvider(): AIProvider {
  const name = (process.env.AI_PROVIDER ?? 'openai').toLowerCase()
  switch (name) {
    case 'openai':    return new OpenAIProvider()
    case 'anthropic': return new AnthropicProvider()
    case 'gemini':    return new GeminiProvider()
    case 'mock':      return new MockProvider()
    default: throw new Error(`Unknown AI_PROVIDER: ${name}`)
  }
}
```

**Shared behaviour across providers:**
- One canonical system prompt (Korean-language songwriting expert persona; instructs original creative output only; specifies JSON schema with exactly 10 songs).
- User prompt: structured serialisation of `Selections` (one line per category, custom inputs appended).
- All providers must return JSON conforming to `GenerationResult` schema (minus `generatedAt`/`provider` which the API layer fills in).
- On JSON parse failure: retry once with stricter instruction; then surface `INVALID_RESPONSE` error.

**Provider-specific configuration:**
- **OpenAI:** `response_format: { type: 'json_object' }`.
- **Anthropic:** tool-use with a `submit_playlist` tool that has the schema; cache the system prompt with `cache_control` for cost.
- **Gemini:** `responseMimeType: 'application/json'` + `responseSchema`.
- **Mock:** returns deterministic dummy data after a 600 ms delay.

## 9. UI components

**`OptionSection`** — unified component used 10× with different props:
- Props: `number`, `emoji`, `title`, `subtitle`, `options[]`, `selected`, `onToggle`, `multi`, `customInput`, `onCustomChange`, `placeholder`.
- Renders: numbered heading, button grid (CSS grid `auto-fill`, gap 8px), free-text input below.

**`OptionButton`** — toggle button. Tailwind variants for selected (filled accent colour) vs unselected (outlined). Wraps emoji + label.

**`LengthSlider`** — controlled `<input type=range min=1 max=10 step=1>` with current value badge.

**`ResultPanel`** — appears below action buttons after generation:
- Top: composite prompt in a monospace box with copy button.
- Below: responsive grid of 10 song cards (title + concept, each with copy button).
- During loading: 10 skeleton cards.

**`HistoryDrawer`** — right-side slide-over (Tailwind `transform`/`transition`):
- Lists past entries (title preview + provider tag + timestamp).
- Click → loads the entry into `ResultPanel`.
- Clear-all button.

**Page layout (`app/page.tsx`):**
- Minimal left sidebar (brand name only — no full nav from original).
- Centre column: scrollable option sections + sticky-ish action button row at bottom of the section list.
- Result panel renders inline below action buttons; auto-scrolls into view on completion.

## 10. Static preset data

`lib/options.ts` exports arrays for each of the 10 sections, e.g.:
```ts
export const GENRES = [
  { emoji: '🎹', label: '시티팝(City Pop)' },
  { emoji: '🎧', label: '로파이(Lo-fi)' },
  // ... ~34 entries, matching the original page's preset list
]
```
Single source of truth for both the UI and the prompt builder.

## 11. Error handling

| Code | Trigger | UI behaviour |
| --- | --- | --- |
| `EMPTY_SELECTION` | Zero options across all sections | Client-side toast: "최소 1개 옵션을 선택해주세요" |
| `MISSING_API_KEY` | Provider env key empty | Toast: "환경변수에 {PROVIDER}_API_KEY를 설정해주세요" |
| `LLM_ERROR` | Network / upstream API failure | Toast with upstream message |
| `INVALID_RESPONSE` | JSON parse fails twice | Toast: "AI 응답을 해석할 수 없습니다. 다시 시도해주세요" |

API route response shape on error:
```json
{ "error": { "code": "MISSING_API_KEY", "message": "..." } }
```

Generate button is disabled while a request is in flight.

## 12. Testing strategy

- **Mock provider** is the primary dev/test tool — exercise UI, loading, history, error toasts without burning API quota.
- **Manual smoke test** with each real provider once per major change (capture screenshots).
- No automated unit tests in MVP (scope decision — yagni).

**Acceptance checklist (post-implementation):**
- [ ] All 10 sections render with correct presets and toggle behaviour.
- [ ] Single-select sections (BPM/age/language) enforce single choice.
- [ ] Custom-input textboxes work and are sent to the LLM.
- [ ] Length slider moves smoothly 1↔10 with live label.
- [ ] Mock provider returns a result panel with 10 song cards.
- [ ] At least one real provider returns a valid `GenerationResult` end-to-end.
- [ ] localStorage history stores up to 20 entries and reloads on click.
- [ ] All 4 error codes show appropriate toasts.
- [ ] API key never appears in browser-side bundles or network requests.
